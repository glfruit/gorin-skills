# Pipeline/Orchestrator 创建经验

## 核心模式

**数据流要可视化**：
- SKILL.md 里画 ASCII 流程图，标注每步输入/输出
- 每步的失败处理路径也要画出来
- agent 看到流程图才知道整体结构

**幂等设计**：
- cron 触发的 pipeline 必须幂等（重复运行不出错）
- 用状态文件/flag 文件标记进度，支持断点续跑
- 不要假设"上次已经跑过了"

**可观测性**：
- 每步输出日志（`>&2` 或 `--verbose`）
- 关键步骤有状态报告（成功/跳过/失败 + 原因）
- cron job 失败时通知到人

## 多源聚合

- 每个源独立处理，一个源挂了不影响其他源
- 结果汇聚时要有去重和优先级
- 最终输出统一格式，不管中间经过多少步

## 常见陷阱

| 陷阱 | 表现 | 解决 |
|------|------|------|
| 非幂等 | cron 重跑产生重复数据 | 状态文件 + 去重 |
| 单点失败 | 一个步骤失败整条 pipeline 中断 | 每步 try/catch，失败降级 |
| 状态丢失 | 中断后从头跑 | 检查点/状态文件 |
| 日志不足 | 出了问题不知道哪步坏了 | 每步输出 status |
| 依赖链过长 | A→B→C→D 一步挂全挂 | 尽量扁平化，减少串行依赖 |

## 验证清单

- [ ] 流程图在 SKILL.md 中
- [ ] 每步有独立的错误处理
- [ ] 重复运行安全（幂等）
- [ ] 有状态检查/断点续跑能力
