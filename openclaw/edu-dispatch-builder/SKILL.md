# edu-dispatch-builder

## Core Principles
- 写手不需要自己找信息，派发时必须把当前板块所需输入一次性给全。
- 派发指令必须自包含，至少覆盖板块规格、教学设计约束、上下文、格式要求、禁止项、交付要求。
- 禁止省略上下文，尤其不能只给一行大纲就让写手自行猜测课程结构或前后衔接。

## When to Use
- 用于 Phase 2 中，每个板块正式派发给 edu-content 子代理之前。
- 用于根据 Excel 大纲行生成可直接复制的完整派发文本。
- 不负责实际执行派发。
- ❌ 真正创建子代理并发出任务，由 edu-tl 使用 sessions_spawn 完成。

## Workflow
1. 读取 Excel 大纲中的目标板块规格行。
2. 读取 `teaching-design.md`，提取教学框架、技术边界、AI助学定义和其他关键设计约束。
3. 读取已有手稿，提取当前板块的前置上下文和后续边界。
4. 按固定模板构造完整派发指令，确保写手无需二次检索。

## 固定指令结构模板

```markdown
## 派发指令: [板块名]

### 基本信息
- 课程: [课程名]
- 项目: [项目名]
- 任务: [任务名]
- 板块: [模块/板块名]
- 大纲行号: [行号]

### 板块规格
[从 Excel 提取的具体内容概要]

### 教学设计约束
[从 teaching-design.md 提取的约束列表]

### 上下文
#### 前置板块（已完成的）
[前置板块的标题和关键内容摘要，帮助写手理解上下文]

#### 后续板块（待完成的）
[后续板块的标题，帮助写手控制内容边界]

### 格式要求
[从治理方案提取的格式规范]

### 禁止项
- 不使用"了解/掌握/熟悉/培养/提高"等虚词
- 不添加 TBD/待定/TODO
- 不在拓展知识板块写 AI 前沿技术（只写专业拓展知识）
- 不修改已完成的板块内容
- [其他项目特定禁止项]

### 自检清单
写完后自查：
1. 知识点是否全部覆盖？
2. 教学目标动词是否合规？
3. 是否有占位符？
4. 板块间过渡是否自然？

### 交付
- 输出文件: [目标文件路径]
- 输出格式: Markdown
```

## Notes
- `scripts/dispatch_builder.py` 提供单板块和批量构造两种入口。
- 若项目治理文档存在，应优先吸收 `delivery-spec.md`、`content-architecture.md`、`framework-decision.md`、`requirements.md` 中的约束，再回填到派发指令中。
- 若手稿尚未形成，可显式写明“暂无已完成板块手稿”，但不能省略前后结构边界。
